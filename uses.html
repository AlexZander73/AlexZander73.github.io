Uses

Tools I keep reaching for

The common thread is durability. I prefer tools that can survive travel, handoffs, quiet months, and low-maintenance publishing without becoming a project of their own.

Daily Build Environment

Platform

macOS, Git, VS Code

A simple local setup with Git at the center. I want the filesystem, repo state, and deployment path to stay legible.

Publishing

GitHub + GitHub Pages

For personal surfaces, app support pages, and static documentation, Pages gives me durability without another backend to nurse.

Web Stack

Plain HTML, CSS, and JS

I use frameworks when they earn their place. For this site and adjacent product surfaces, static-first wins on speed and upkeep.

AI and Automation

Local-first AI workflow

  • Scope narrowly. I use local models for summarisation, retrieval, and small decision-support loops, not vague magic demos.
  • Keep outputs checkable. Fast verification matters more than flashy generation.
  • Stay private by default. If the task involves personal notes or sensitive drafts, local-first is the default posture.

Automation rules

  • Only automate the boring parts. Syncing catalogs, publishing static assets, and reducing repeated setup all qualify.
  • Prefer small scripts. I trust a readable file and a single-purpose workflow more than a sprawling platform.
  • Graceful failure is mandatory. If an automation breaks, the underlying site or product still needs to work.

Product Workflow

  • Static marketing and support. Public support, privacy, and product pages live comfortably as static files when the structure is clear.
  • JSON where it helps. Small data files are enough for projects, changelogs, and catalogs without dragging in a CMS.
  • Markdown for longer writing. Blog posts and reference notes belong in simple source formats that stay portable.
  • Godot for feel studies. When I want to explore interaction or atmosphere, I'd rather prototype quickly than over-spec the idea.

Field Kit

I'm biased toward setups that still make sense away from a perfect desk: offline notes, small files, dependable publishing, and products that keep working even if the internet gets flaky. That bias shows up in how I write, prototype, and ship.
